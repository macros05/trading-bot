### 2 · Bind-mount sync bug — diagnosis + fix (commit `8ad30c3`)
**Symptom:** `data/bot_health.json` on the host showed `last_close=100.0, rsi=NaN` for 19 minutes while inside the container the file held `last_close=81400, rsi=46.4`. Dashboard was reading the stale host view.

**Root cause:** Docker single-file bind mounts on overlayfs sever the inode link the first time `O_TRUNC` is applied. `Path.write_text(data)` opens with `O_WRONLY|O_CREAT|O_TRUNC`. After the first severance:
- Container's `/app/data/bot_health.json` inode = 281995
- Host's `/root/trading-bot/data/bot_health.json` inode = 281997 (frozen at the moment of severance)
- All subsequent writes inside the container go to inode 281995; host never sees them.

**Two-pronged fix:**
1. **`/root/docker-compose.yml`** — switched `trading_bot` and `trading_api` from single-file bind mounts to a directory bind mount (`/root/trading-bot/data:/app/data`). Atomic write via `os.replace` now works (rename within the same dir). **This file is not under git — change persists on the host only.**
2. **`core/loop.py::_atomic_or_direct_write` and `core/state.py::StateManager._persist`** — replaced the `Path.write_text` fallback with `os.O_WRONLY|os.O_CREAT` + `os.ftruncate(fd, 0)` + `os.write`. Preserves the inode even if someone later reverts to single-file mounts. Defense in depth.

**Tests added:** `tests/test_atomic_write.py` (7 cases) — happy path, EBUSY fallback, EXDEV fallback, other-errno re-raise, inode preservation, truncation of longer prior content, and `StateManager._persist` fallback.

**Verified:** after rebuild, host inode == container inode (281997), and the host's view updates every tick.
